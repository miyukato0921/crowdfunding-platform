# 認証まわりのメモ

## ① ログイン処理はどのファイルの何行目にある？

**ファイル:** `app/api/admin/login/route.ts`

| 行 | 内容 |
|----|------|
| 8 | リクエストからメール・パスワードを取り出す |
| 17 | DBでメールアドレスを検索してユーザーを探す |
| 30〜38 | bcryptでパスワードを照合（平文の場合は自動でbcryptに昇格） |
| 47 | ランダムなトークンを生成（48バイト・hex文字列） |
| 48 | セッションの有効期限を7日後に設定 |
| 50〜53 | トークンをDBの `admin_sessions` テーブルに保存 |
| 56〜63 | トークンをCookie（`admin_session`）に入れてブラウザに返す |

---

## ② 「ログインしていないとアクセスできない」はどうやって実現している？

**仕組みは2段階。**

### 段階1 — セッション検証関数（`lib/auth.ts` 6〜19行目）

`getAdminSession()` という関数がある。呼ばれると：

1. ブラウザのCookieから `admin_session` トークンを取り出す
2. DBの `admin_sessions` テーブルでそのトークンを検索する
3. 見つかって期限内なら → ユーザー情報を返す
4. 見つからない・期限切れなら → `null` を返す

### 段階2 — 各ページで呼び出す

管理画面の各ページ（`app/admin/(protected)/` 配下）で `getAdminSession()` を呼び、`null` だったら `/admin/login` にリダイレクトする。

```ts
// 例: app/admin/login/page.tsx 7〜8行目
const session = await getAdminSession()
if (session) redirect("/admin")   // ← ログイン済みなら管理画面へ
```

ルートグループ `(protected)` のレイアウトで一括チェックしている場合も同様の仕組み。

## ③ ログアウト処理は何をしている？

**ファイル:** `app/api/admin/logout/route.ts`

順番に2つのことをしている：

1. **DBからトークンを削除**（10行目）
   - `admin_sessions` テーブルから該当トークンのレコードをDELETE
   - これにより、古いトークンが残っていても使えなくなる

2. **ブラウザのCookieを削除**（14行目）
   - `admin_session` Cookieを空にしてブラウザに送り返す
   - これにより、ブラウザ側にもトークンが残らなくなる

**なぜ両方削除するのか？**
- DBだけ消す → Cookieが残るので別PCで使い回せてしまう
- Cookieだけ消す → DBにトークンが残り続ける（無駄なレコードが溜まる）
- 両方消して初めて完全にセッションが終了する
